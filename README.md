# Habit Tracker Bot

Telegram-бот для трекинга привычек с backend-сервисом на FastAPI.

## Стек

- FastAPI
- PostgreSQL
- SQLAlchemy
- Alembic
- PyTelegramBotAPI
- Docker Compose
- Poetry
- JWT

## Запуск

1. Создать файл `.env` на основе `.env.template`.

2. Указать токен Telegram-бота:

```env
BOT_TOKEN=your_telegram_bot_token
JWT_SECRET_KEY=your_random_secret_key
```

`JWT_SECRET_KEY` должен быть случайной длинной строкой. Его можно сгенерировать командой:

```bash
python -c "import secrets; print(secrets.token_urlsafe(64))"
```

3. Запустить проект:

```bash
docker compose up -d --build
```

4. Применить миграции:

```bash
docker exec -it habits_backend alembic upgrade head
```

5. Открыть Swagger:

```text
http://127.0.0.1:8000/docs
```

## Backend

Проверка сервиса:

```text
http://127.0.0.1:8000/health
```

## Команды бота

- `/start` — регистрация и авторизация пользователя
- `/habits` — список привычек
- `/add_habit` — добавить привычку
- `/complete_habit` — отметить выполнение привычки
- `/skip_habit` — отметить невыполнение привычки
- `/edit_habit` — изменить привычку
- `/delete_habit` — удалить привычку

## Готовность проекта по ТЗ

На текущий момент реализованы следующие компоненты:

### Backend (FastAPI)

- JWT-аутентификация пользователей;
- регистрация пользователей;
- получение JWT-токена;
- CRUD-операции для привычек;
- отметка выполнения и пропуска привычек;
- PostgreSQL в качестве основной базы данных;
- SQLAlchemy ORM;
- Alembic для миграций;
- Docker Compose для запуска сервисов.

### Telegram Bot

Реализованы команды:

- `/start` — регистрация и авторизация пользователя;
- `/help` — список доступных команд;
- `/habits` — просмотр привычек;
- `/add_habit` — создание привычки;
- `/complete_habit` — отметить выполнение;
- `/skip_habit` — отметить невыполнение;
- `/edit_habit` — изменить привычку;
- `/delete_habit` — удалить привычку.

Архитектура Telegram-бота:

```text
bot/
├── main.py              # точка входа
├── handlers/            # обработчики команд
│   ├── common.py
│   └── habits.py
├── config/              # настройки приложения
├── services/            # работа с backend API и бизнес-логика
```

### Notifier

Реализован отдельный сервис уведомлений, запускаемый как отдельный контейнер Docker.

## Аутентификация в Swagger

Для аутентификации используется JWT.

Пользователь:

1. Регистрируется через эндпоинт `/auth/register`;
2. Получает JWT-токен через `/auth/token`;
3. В Swagger нажимает кнопку **Authorize**;
4. Передаёт токен в формате:

```text
Bearer <access_token>
```

Все защищённые эндпоинты используют зависимость `get_current_user`.

---

## Аутентификация пользователя из Telegram-бота

При выполнении команды `/start`:

1. Бот получает `telegram_id` и `username`;
2. Формирует служебный пароль;
3. Выполняет запрос на `/auth/register`;
4. Получает JWT-токен через `/auth/token`;
5. Сохраняет токен в собственной базе данных бота.

Если пользователь уже зарегистрирован, повторная регистрация не выполняется.

---

## Хранение токенов в Telegram-боте

У бота используется отдельная SQLite база данных.

Таблица `user_tokens` хранит:

- `telegram_id`;
- `username`;
- служебный пароль;
- `access_token`.

Токены не хранятся в памяти приложения и не теряются после перезапуска контейнера.

---

## Обновление токенов

Для работы с backend используется отдельный сервис.

При обращении к API:

1. Бот получает JWT-токен из SQLite.
2. Выполняет запрос к backend.
3. При необходимости бот может получить новый JWT через эндпоинт `/auth/token`.
4. Новый токен сохраняется в SQLite и используется для последующих запросов.

## Взаимодействие сервисов

Проект состоит из четырёх сервисов:

### 1. Telegram Bot

Отвечает за взаимодействие с пользователем.

### 2. Backend (FastAPI)

Предоставляет REST API и выполняет бизнес-логику.

### 3. PostgreSQL

Хранит данные пользователей, привычек и истории выполнения.

### 4. Notifier

Отдельный сервис для отправки уведомлений.

### Последовательность взаимодействия

1. Пользователь отправляет команду в Telegram.
2. Telegram Bot принимает сообщение.
3. Bot выполняет запрос к Backend API.
4. Backend обращается к PostgreSQL.
5. PostgreSQL возвращает данные.
6. Backend формирует ответ.
7. Bot отправляет ответ пользователю.
8. Notifier периодически обращается к Backend и отправляет уведомления пользователям.
